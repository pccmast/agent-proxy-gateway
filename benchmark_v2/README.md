# Agent Proxy Gateway — Phase 2 压测计划 v2（真实 GPU 后端）

> **版本**: v2.0  
> **制定日期**: 2026-06-12  
> **前置实验**: Phase 1 基准压测已完成（mock upstream，数据见 `benchmark/results/`）  
> **后端**: 本地 GPU（GTX1650 4GB）+ lm studio + minicpm-v-4.6 Q8_0（1.92G）  
> **约束**: 不修改原 `benchmark/` 文件；不做长时间压测（总时长 < 15 分钟）

---

## 1. 为什么需要 Phase 2？

Phase 1 用 mock upstream（固定 150ms 延迟）测出了网关的**理论上限**（QPS 天花板 8.2，SQLite 瓶颈）。但真实生产环境完全不同：

| 维度 | Phase 1 (Mock) | Phase 2 (真实 GPU) |
|------|---------------|-------------------|
| 延迟分布 | 固定 150ms | 长尾分布（均值 500ms~2s，P95 可能 5s+） |
| 并发模型 | 纯计算，无状态 | 受 GPU VRAM 限制，有排队 |
| 错误类型 | 无 | OOM、KV Cache 溢出、超时 |
| 流式表现 | 固定 9 个 chunk | 真实 token 生成速度 |
| 资源竞争 | 无 | CPU-GPU 内存拷贝、CUDA 上下文切换 |

**Phase 2 的核心问题**：当上游从"完美 Mock"变成"真实慢推理"时，网关的连接池管理、超时策略、熔断逻辑、流式转发是否还能正常工作？

## 2. 实验设计（保守策略）

### 2.1 硬件约束分析

- **GPU**: GTX1650（4GB VRAM，无 Tensor Core，Compute Capability 7.5）
- **模型**: minicpm-v-4.6 Q8_0（1.92G 参数，量化后约 2.4GB 内存占用）
- **可用 VRAM**: 4GB - 2.4GB（模型）≈ 1.6GB 空闲
- **并发估算**: 每个请求需要 KV Cache（与 prompt 长度成正比），保守估计同时服务 3-5 个并发请求
- **推理速度**: 约 5-15 tokens/s（GTX1650 的 FP16 性能有限，Q8_0 用 INT8 会快一些）

### 2.2 实验矩阵

| 实验 | 并发 | 请求数 | 模式 | 时长 | 目的 |
|------|------|--------|------|------|------|
| **E1: 基线** | 1 | 5 | 非流式 | ~2-5 min | 测量真实 GPU 推理延迟基线 |
| **E2: 轻并发** | 3 | 10 | 非流式 | ~3-5 min | 观察 3 并发下 GPU 排队情况 |
| **E3: 阶梯负载** | 5→10→20 | 各 10 | 非流式 | ~5-8 min | 发现性能拐点 |
| **E4: 流式基线** | 1 | 5 | 流式 | ~2-4 min | 测量真实流式 TTFT |
| **E5: 流式并发** | 3 | 6 | 流式 | ~3-5 min | 观察流式并发下的 chunk 间隔 |

**不做的事**：
- 不做 50+ 并发（GPU 会 OOM 或严重排队）
- 不做 30 分钟长稳测试（用户要求）
- 不修改网关代码（新增可观测性只在客户端侧记录）

## 4. 可观测性策略（简化版）

由于不修改网关代码，可观测性在**客户端侧**实现：

```
客户端侧记录指标：
├── t0: 请求开始时间
├── t1: 连接建立时间（TCP + TLS）
├── t2: 首字节时间（TTFB）
├── t3: 完整响应时间
├── 流式额外记录：
│   ├── first_chunk_time: 首 chunk 到达时间
│   ├── chunk_intervals: 相邻 chunk 间隔
│   └── total_tokens: 预估 token 数（从 response 解析）
```

**关键洞察**：
- `t2 - t1` ≈ 网关处理时间 + 模型首 token 时间（TTFT）
- `t3 - t2` ≈ 剩余 token 生成时间
- 流式场景下 `chunk_intervals` 反映真实 token 生成速度

## 5. 风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| GPU OOM | 中 | 并发超过 5 时触发，实验设计中 E3 阶梯到 20 可能触发；如果 OOM 则提前终止并发级别，记录为"拐点" |
| lm studio 崩溃 | 低 | 每次实验前调用 `/health` 检查 |
| 请求超时 | 高 | 网关默认 120s 超时，真实 GPU 推理慢请求可能超时；记录超时率作为关键指标 |
| 结果波动大 | 高 | GPU 推理延迟受温度、后台进程影响；每个实验跑 3 次取中位数 |

## 6. 面试话术预设（实验前准备）

> "Phase 1 用 mock upstream 测了网关的理论性能，发现 SQLite 写入锁是瓶颈（QPS 天花板 8.2）。Phase 2 用真实 GPU 后端（GTX1650 + minicpm-v-4.6）验证网关在生产级负载下的表现。真实 GPU 的长尾延迟（500ms~5s）会测试网关的连接池管理、超时策略和熔断逻辑。由于 GPU VRAM 限制，并发控制在 20 以内，重点观察错误率和延迟拐点，而不是追求高 QPS。"
